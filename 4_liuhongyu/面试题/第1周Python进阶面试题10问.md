## 1. `self` 的本质是什么？为什么实例方法第一个参数必须是 `self`？

**参考答案**：`self` 代表调用该方法的实例对象本身。Python 调用 `obj.method(args)` 时，解释器自动将其转换为 `ClassName.method(obj, args)`，所以实例方法的第一个参数必须接收这个实例引用。叫 `self` 只是约定，叫别的名也行，但不推荐。

## 2. 类属性、实例属性、私有属性三者的区别和访问优先级是什么？

**参考答案**：

| 类型 | 定义方式 | 作用域 | 访问方式 |
|------|---------|--------|---------|
| 类属性 | 类体内直接定义 | 所有实例共享 | `ClassName.attr` 或 `self.attr` |
| 实例属性 | `self.xxx` 在 `__init__` 中定义 | 每个实例独立 | `self.xxx` |
| 私有属性 | `self.__xxx` | 名称改编为 `_ClassName__xxx` | 类内部访问 |

优先级：实例属性 > 类属性。修改类属性应通过 `ClassName.attr`，否则会意外创建同名实例属性遮蔽类属性。

## 3. `ABC` + `@abstractmethod` 和手动抛 `NotImplementedError` 有什么区别？各适合什么场景？

**参考答案**：

| 方案 | 检测时机 | 优点 | 缺点 |
|------|---------|------|------|
| `NotImplementedError` | 运行时（调用才报错） | 轻量、灵活 | 不调用不报错 |
| `ABC + @abstractmethod` | 实例化时（立即报错） | 编译时强制检查 | 稍重 |

小项目或插件接口用 `NotImplementedError`，框架级公共接口用 `ABC`。

## 4. `super()` 调的一定是直接父类吗？结合 MRO 解释。

**参考答案**：不一定。`super()` 按 MRO（Method Resolution Order，C3 线性化）查找下一个类，而不是简单调用直接父类。多重继承时尤其明显：

```python
class A: pass
class B(A): pass
class C(A): pass
class D(B, C): pass

# D.__mro__ → D → B → C → A → object
# B 中的 super() 会跳到 C，而不是 A
```

## 5. Python 多线程受 GIL 限制，为什么 I/O 密集场景仍然能加速？

**参考答案**：GIL 约束的是同一时刻只能有一个线程执行 Python 字节码。但线程在等待 I/O（网络请求、文件读写、`time.sleep`）时会主动释放 GIL，让其他线程获得执行机会。因此 I/O 密集场景多线程有效；纯 CPU 计算场景多线程无效，应改用 `multiprocessing`。

## 6. 函数装饰器和类装饰器的核心差异是什么？什么时候必须用类装饰器？

**参考答案**：

| 维度 | 函数装饰器 | 类装饰器 |
|------|----------|----------|
| 状态存储 | 依赖闭包 + `nonlocal` | 靠 `self.xxx` 自然持久 |
| 外部交互 | 难以暴露方法 | 可扩展 `clear()` / `get_count()` 等 |
| 传参 | 三层嵌套 | `__init__` 收配置，`__call__` 收函数 |

需要持久保存状态（缓存、计数、限流时间窗口）或需要外部主动操控内部数据时，必须用类装饰器。

## 7. 带参数的类装饰器执行流程是怎样的？写出 `@Logger("DEBUG")` 等价调用。

**参考答案**：

```python
@Logger("DEBUG")
def login(name): ...

# 等价于：
login = Logger("DEBUG").__call__(login)
```

执行顺序：
1. `Logger("DEBUG")` → `__init__` 存储 `self.level = "DEBUG"`
2. `.__call__(login)` → 接收函数并返回 `wrapper`
3. `login` 实际指向 `wrapper`

## 8. `async/await` 和多线程都是并发，本质区别是什么？分别适合什么场景？

**参考答案**：

| 维度 | 多线程 | 异步 |
|------|--------|------|
| 切换者 | 操作系统抢占式调度 | 协程主动让出（`await`） |
| 内存 | 每线程 ~MB | 每协程 ~KB |
| 数据安全 | 需要 Lock | 单线程按顺序，天然安全 |
| 适合 | I/O 等待 + 少量计算 | 大量 I/O 等待（网络/API） |

简单区分：任务需要改同一份共享数据 → 异步更安全；已有大量同步代码想提速 → 多线程改造成本低。

## 9. 用 `asyncio.gather` 并发执行 4 个任务（2 个 1s + 2 个 2s），总耗时多少？为什么？

**参考答案**：约 2 秒。`gather` 同时启动所有协程，总耗时等于其中最长的单个任务耗时（2s），而非累加（6s）。异步的核心优势就在于此——等待期间不阻塞，切换去执行其他就绪的任务。

## 10. `Lock` 在什么情况下必须用？不用会出现什么问题？`with lock:` 比手动 `acquire/release` 好在哪里？

**参考答案**：多线程同时写共享可变数据（如 list append / dict update / 计数器自增）时必须加锁。不加锁会导致竞态条件：两个线程同时读到同一个值，各自 +1 后写回，最终只 +1 而非 +2，数据丢失。

`with lock:` 优势：
- 自动 `acquire()` 和 `release()`，即使内部抛异常也能安全释放
- 不会因忘记 `release()` 导致死锁
- 代码简洁，意图清晰
