## 项目概述

一个完整的奶茶门店收银系统，不同饮品类型有各自的优惠规则：

| 饮品类型 | 文件 | 优惠规则 |
|----------|------|----------|
| 饮品基类 | `base_drink.py` | 定义统一接口 + 全场折扣 |
| 果茶 | `fruit_tea.py` | 全场折扣基础上额外 95 折 |
| 奶盖茶 | `milk_cap.py` | 买 2 杯及以上立减 3 元 |
| 咖啡 | `coffee.py` | 全场折扣基础上统一 88 折 |

系统还支持订单文件持久化、多线程安全售卖、低价饮品筛选等功能。

## 技术要点深度解读

### OOP 四大方法体系

`base_drink.py` 中完整展示了 Python 类的四种方法类型：

| 方法类型 | 装饰器 | 首参数 | 操作对象 | 本项目实例 |
|----------|--------|--------|----------|------------|
| 实例方法 | 无 | `self` | 单个实例属性 | `sell()`, `get_stock()`, `print_ticket()` |
| 类方法 | `@classmethod` | `cls` | 类属性（全类共享） | `set_shop_discount()` |
| 静态方法 | `@staticmethod` | 无 | 独立工具计算 | `check_drink_name()` |
| 抽象方法 | `@abstractmethod` | `self` | 子类必须重写 | `get_final_price()` |

**实例方法与类方法的核心区别**：

```python
# 实例方法：操作单个饮品数据
mango = FruitTea("杨枝甘露", 16)
mango.sell(2)           # 只影响 mango 这一杯的库存

# 类方法：影响所有饮品
BaseDrink.set_shop_discount(0.9)  # 所有饮品的折扣都变为 0.9
```

**抽象方法的设计意义**：`get_final_price()` 在基类中只有声明没有实现（`pass`），子类必须重写。这是一种接口契约——所有饮品都必须有自己的计价逻辑，但规则各不相同。如果子类忘了重写，实例化时直接报 `TypeError`，符合 Fail Fast 原则。

**静态方法**：`check_drink_name()` 不依赖任何实例或类数据，只是一个独立的校验工具，挂在 `BaseDrink` 下纯粹是命名空间的逻辑归属。

### 继承与方法重写（多态）

三种饮品子类各自重写 `get_final_price()`，实现的优惠规则完全不同：

```python
# 果茶：额外 95 折
class FruitTea(BaseDrink):
    def get_final_price(self, buy_num):
        return round(self.price * buy_num * self.shop_discount * 0.95, 2)

# 奶盖茶：买 2 杯立减 3 元
class MilkCapTea(BaseDrink):
    def get_final_price(self, buy_num):
        origin = self.price * buy_num * self.shop_discount
        return round(origin - 3 if buy_num >= 2 else origin, 2)

# 咖啡：统一 88 折
class Coffee(BaseDrink):
    def get_final_price(self, buy_num):
        return round(self.price * buy_num * self.shop_discount * 0.88, 2)
```

**多态的威力**体现在 `checkout_batch()` 中：

```python
def checkout_batch(drink_list):
    for drink in drink_list:
        price = drink.get_final_price(1)  # 无需判断 drink 的具体类型
```

无论 `drink` 是 FruitTea、MilkCapTea 还是 Coffee，`get_final_price()` 都会自动路由到对应子类的实现。这就是里氏替换原则——父类出现的地方，子类可以无缝替换。

**子类属性扩展**：FruitTea 新增了 `self.type = "果茶"`，MilkCapTea 新增了 `self.__milk_cap_cost` 和 `get_milk_cap_cost()`，Coffee 则完全复用父类属性。各子类可以根据业务需要自由扩展。

**`super().__init__()` 的三种模式**：

```python
# 模式 1：透传父类参数
class FruitTea(BaseDrink):
    def __init__(self, name, price):
        super().__init__(name, price)  # 用户传入 name
        self.type = "果茶"

# 模式 2：子类固定父类参数
class FixedTea(BaseDrink):
    def __init__(self):
        super().__init__("固定名称", 10)

# 模式 3：子类扩展参数
class ExTea(BaseDrink):
    def __init__(self, name, price, sweet):
        super().__init__(name, price)
        self.sweet = sweet
```

忘记调用 `super().__init__()` 会导致父类的 `self.name`/`self.price` 未初始化，后续访问直接 `AttributeError`。

### with 上下文管理器

本项目在两处使用 `with`，展示了它统一管理资源的威力：

**文件操作**：

```python
# 传统方式：极易忘记 close() 导致数据丢失
f = open("order.txt", "a", encoding="utf-8")
f.write("珍珠奶茶\n")
f.close()

# with 方式：缩进结束自动关闭
with open("order.txt", "a", encoding="utf-8") as f:
    f.write("珍珠奶茶\n")
```

**线程锁**：

```python
# 传统方式：必须 try/finally 防止死锁
lock.acquire()
try:
    global_stock["珍珠奶茶"] -= 1
finally:
    lock.release()

# with 方式：自动 acquire + release
with stock_lock:
    global_stock["珍珠奶茶"] -= 1
```

两种场景本质相同——管理需要"获取 → 使用 → 释放"模式的资源。`with` 的背后是 `__enter__` 和 `__exit__` 协议：进入 `with` 块时调用 `__enter__`（打开文件/获取锁），离开时调用 `__exit__`（关闭文件/释放锁），即使发生异常也能保证资源释放。

### __init__.py 包管理

**核心认知**：`__init__.py` 与 `def __init__(self)` 是两个完全不同的概念。

| | `__init__.py` | `def __init__(self)` |
|---|---|---|
| 位置 | 文件夹内的文件 | class 内部的方法 |
| 作用 | 标记文件夹为 Python 包 | 创建对象时自动执行的构造方法 |
| 调用时机 | import 时 | 实例化时 |

**两种使用方式**：

```python
# 方式 1：空白文件（课堂标准写法）
# goods/__init__.py 无任何代码
# main.py 中使用完整路径导入
from goods.base_drink import BaseDrink
from goods.fruit_tea import FruitTea

# 方式 2：预导入（本项目采用）
# goods/__init__.py
from goods.base_drink import BaseDrink
from goods.fruit_tea import FruitTea
from goods.milk_cap import MilkCapTea
from goods.coffee import Coffee

# main.py 中简写
from goods import BaseDrink, FruitTea, MilkCapTea, Coffee
```

**缺少 `__init__.py` 的后果**：直接报 `ModuleNotFoundError: No module named 'goods'`。解决方式：在 goods 文件夹新建空白 `__init__.py`。

### 列表推导式

```python
def get_cheap_drinks(drink_dict, limit):
    cheap_data = {name: p for name, p in drink_dict.items() if p < limit}
    return list(cheap_data.keys())

menu = {"珍珠奶茶": 12, "杨枝甘露": 16, "芝士葡萄": 15, "美式咖啡": 10}
get_cheap_drinks(menu, 14)  # ['珍珠奶茶', '美式咖啡']
```

这里使用了字典推导式 `{key: value for ... if ...}`——一次性完成"筛选 + 构造"两步操作。相比传统 for 循环，代码更简洁且性能更好（CPython C 级别实现）。

### 生成器

```python
def order_record_generator(order_list):
    for order in order_list:
        yield f"饮品：{order[0]}，数量：{order[1]}，总价：{order[2]}"
```

`yield` 将函数转为生成器。每次调用 `next()` 或 `for` 迭代时才计算下一个值，而非一次性生成全部结果。

**内存优势**：如果订单列表有百万条，普通方式需要一次性将所有格式化字符串加载到内存；生成器每次只在内存中保存一条。这也是 AI Agent 流式输出（逐 token 返回而非等全部生成完）的底层原理。

### 多线程与线程安全

```python
global_stock = {"珍珠奶茶": 50, "杨枝甘露": 30}
stock_lock = threading.Lock()

def sell_drink_thread_safe(drink_name, sell_num):
    with stock_lock:
        if global_stock[drink_name] < sell_num:
            return False
        global_stock[drink_name] -= sell_num
        return True
```

**为什么需要 Lock**：`global_stock[drink_name] -= 1` 看似一行代码，实际分三步：读取当前库存 → 减 1 计算 → 写回。两个线程同时执行可能读到相同的旧值，导致"超卖"——两个线程各卖了 1 杯但库存只减了 1。

`with stock_lock` 保证同一时刻只有一个线程能进入临界区，彻底杜绝竞态条件。
